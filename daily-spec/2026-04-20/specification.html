MDPipeline Specification

Codebase-grounded scientific review of the actual current shipped MDPipeline surfaces: CLI, one-pose selection policy, preparation/protonation/tooling layers, OpenMM MD, MM/GBSA analysis, output contract, and major scientific limitations.

Actual shipped CLI surface

CommandCurrent code-grounded role
check-openmmOpenMM import/platform/context preflight; the only command that does not require --run-dir.
selectSelection only; writes selected_model.json, tool_config.json, and protocol.json.
prepareSelection + pose validation + prep; writes prep_summary.json.
simulatePrep plus MD; writes md_summary.json.
analyzeReuses prep/MD outputs from existing summaries and runs trajectory analysis + MM/GBSA.
fullRuns the end-to-end per-case pipeline.

Selection policy actually implemented

  • Model selection is strictly lowest score_only_affinity_kcal_per_mol from upstream TSV.
  • The batch contract is rigidly pinned to n_cifs_total = 10.
  • Manifest decision must be run and ligand identity must match between manifest and result row.
  • CIF/model paths are reconstructed from run_dir and model_index rather than trusting stale TSV paths.

This means the current scientific commitment is not a consensus or multi-pose policy; it is one-pose selection on one upstream scalar.

Preparation, protonation, and parameterization surfaces

  • Default ligand backend is GAFF2/BCC via antechamber + parmchk2.
  • Alternate backend is openff-sage, but it requires extra packages and still preserves Amber-compatible downstream outputs.
  • Protein protonation modes are only none and propka-pdb2pqr.
  • Ligand protonation modes are only none and deterministic single-state epik.
  • Protonation summaries capture command lines, tool paths, and version probes.

MD and MM/GBSA workflow actually shipped

  • MD is OpenMM on Amber prmtop/inpcrd with PME, 1.0 nm cutoff, HBond constraints, and staged restrained equilibration.
  • The default stage schedule ends with production = 20.0 ns.
  • Trajectory analysis is intentionally narrow: RMSD, contacts, minimum distance, and H-bond occupancy summaries.
  • MM/GBSA is run through ParmEd/GROMACS/gmx_MMPBSA with configurable gb/pb, optional decomposition, optional MPI, and entropy off by default.
  • Default MM/GBSA window skips the first 2.0 ns of production.

Output contract and operator evidence

The per-case output contract is real and strongly structured: selected_model.json, tool_config.json, protocol.json, pose_validation.json, prep_summary.json, md_summary.json, trajectory-analysis outputs, MM/GBSA outputs, and a top-level run_summary.json. The tests confirm this structure under shipped code.

Current scientific/operator limitations

  • Single-pose selection: no built-in multi-pose or ensemble ranking surface in the current CLI.
  • Strict ten-model assumption: current code expects a 10-CIF Boltz batch summary.
  • Tool dependence: AmberTools, GROMACS, gmx_MMPBSA, protonation tools, and optional OpenFF packages are all external dependencies.
  • Entropy-off default: default MM/GBSA reporting is not a full free-energy claim.
  • Narrow analytics: no broader conformational/statistical analysis beyond the shipped RMSD/contact/H-bond summaries.
  • Chemistry limits: iodine-containing systems can fail GB decomposition; ligand protonation is single-state only.

Top current clarifications

  • The current shipped CLI is still a per-case six-command surface, not a benchmark or cohort-ranking command layer.
  • Selection remains explicitly one-pose and upstream-score-driven via `score_only_affinity_kcal_per_mol`.
  • The strongest source-backed caution is operational/tool dependence: the pipeline logic is rich, but scientific execution still depends on many external binaries and optional packages.
