MDPipeline Specification for Scientific Review

This version is written primarily for a genetics PhD director and a chemical / molecular engineer focused on thermodynamic interpretation. It still cites code and artifacts, but it foregrounds the scientific purpose, decision relevance, assumptions, and thermodynamic meaning of each step rather than the software architecture alone.

Grounded against current documents and shipped source files. Where the code and the broader scientific ambition diverge, that gap is stated explicitly rather than smoothed over.

Executive view for signoff

QuestionCurrent answer
What does the pipeline currently do?It takes one pre-existing boltz_results_* case, selects one candidate pose, validates file consistency, builds a solvated protein–ligand system, runs restrained MD, and computes MM/GBSA-style outputs for relative ranking.
What is the scientific role?Post-processing and ranking support. Not de novo docking. Not a rigorous absolute binding free-energy system.
What is the biggest scientific simplification?Single-pose decision The shipped selection rule commits to one pose by minimum score_only_affinity_kcal_per_mol.
What is the main thermodynamic caveat?Approximate ranking signal The endpoint is MM/GBSA-style ranking support, not a validated free-energy truth layer.
What is the main operational caveat?External-tool dependent The present host did not have AmberTools / GROMACS / protonation executables on PATH when this page was generated.

What a genetics director should know

  • The pipeline does not explore many biological hypotheses at once; it follows one chosen structural hypothesis per case.
  • Its upstream dependency is a Boltz + Vina result family. That means biological interpretation starts after an upstream structural/pose-generation step has already narrowed the search space.
  • The pipeline's main value is to impose a more physically informed follow-up than raw docking scores alone: explicit solvation, restrained MD relaxation, and MM/GBSA-like scoring.
  • The current shipped operator surface is per-case, not cohort-governed. It is therefore best viewed as a case-processing pipeline, not yet as a program-level decision authority.
  • For signoff, the main strategic question is whether a one-pose, one-run MM/GBSA workflow is acceptable as a decision-support layer for your biology program, or whether replicate / multi-pose / benchmark-governed ranking must be required before use in higher-stakes calls.

What a thermodynamics / molecular engineer should know

  • The workflow contains physically meaningful stages: explicit-solvent system build, restrained equilibration, production MD, trajectory metrics, then MM/GBSA analysis.
  • The current thermodynamic reporting layer is approximate: MM/GBSA with configurable GB/PB solvent models and optional entropy diagnostics, not a formal alchemical free-energy calculation.
  • Default entropy mode is none. This stabilizes reporting but means the default primary outputs are enthalpy-like / component summaries rather than a fully defended binding free energy.
  • Current docs explicitly frame outputs as most useful for relative ranking within a comparable target/ligand series.
  • The present workflow does not, in the shipped CLI, expose a multi-pose ensemble-ranking or benchmark-governed thermodynamic confidence layer.

Scientific workflow

Boltz structural candidates + Vina scoring
  → choose one candidate pose
  → verify that CIF / receptor / ligand files agree
  → prepare receptor and ligand parameters
  → build solvated protein–ligand system
  → run restrained MD and production MD
  → measure structural stability / contacts
  → compute MM/GBSA-style energy summaries
  → write machine-readable outputs

This summary is grounded in MDPIPELINE_SCIENTIST_GUIDE.md, MDPIPELINE_ONE_PAGE_WORKFLOW.md, and the shipped orchestration in md_pipeline/cli.py.

Pose-selection rule

The most important scientific policy in current shipped code is in md_pipeline/selection.py:

  • The pipeline reads batch_results.tsv, batch_manifest.tsv, and batch_summary.json.
  • It chooses the model with the single minimum score_only_affinity_kcal_per_mol.
  • It does not choose by docking_affinity_kcal_per_mol.
  • It validates a ten-candidate assumption through n_cifs_total = 10.
  • If the best choice is ambiguous or required files are missing, it fails closed.

Why this matters scientifically: all downstream MD and MM/GBSA interpretation inherits this one-pose commitment. If the upstream pose ranking is wrong or underdetermined, downstream thermodynamic interpretation is anchored to the wrong structural hypothesis.

Thermodynamic interpretation of the workflow

StageScientific meaningCurrent caution
Pose validationProtects against file mismatches so simulation starts from the intended complex.Integrity check, not a thermodynamic validation.
Receptor / ligand preparationDefines the force-field representation and protonation-sensitive starting state.Strongly tool- and chemistry-dependent.
Solvated system buildMoves from static pose files to an explicit-solvent simulation system.Quality depends on parameterization and preparation assumptions.
Restrained equilibrationAllows limited structural relaxation while preserving gross pose context.Can stabilize a chosen pose without proving it is the biologically dominant pose.
Production MDSamples local dynamics around the selected bound state.Current default is finite and practical, not exhaustive.
Trajectory metricsProvide stability/context signals such as RMSD, contacts, distances, and H-bond occupancy.Supportive evidence, not free-energy proof.
MM/GBSAProvides a tractable energetic ranking approximation from sampled frames.Approximate endpoint; default entropy is off; best interpreted comparatively.

Current default protocol facts

  • ProtocolConfig sets a staged schedule ending with production = 20.0 ns.
  • Default solvent model set is ("gb",).
  • Allowed solvent models are only gb and pb.
  • Allowed entropy modes are none, c2, and ie.
  • Default MM/GBSA window skips the first 2.0 ns of production.

Current scientific limits

  • Single-pose Current shipped code processes one selected pose, not a pose ensemble.
  • Approximate endpoint MM/GBSA is useful but is not equivalent to rigorous free-energy estimation.
  • Per-case surface The shipped CLI is focused on one case at a time, not a benchmark-governed family-ranking release process.
  • Tool dependence Protonation, AmberTools preparation, GROMACS, and gmx_MMPBSA all depend on external executables.
  • Host readiness incomplete On this host, all major scientific executables checked for this page were missing from PATH.

Recommended signoff questions

For the genetics director

  • Is it acceptable that one structural hypothesis is selected before thermodynamic follow-up begins?
  • Are the biological decisions you want to support low enough risk for an MM/GBSA-based ranking layer?
  • Do you require benchmarked cohort-level confidence before this workflow can influence prioritization?

For the thermodynamics / molecular engineer

  • Is the current default 20 ns production schedule enough for the ligand/protein regimes you care about?
  • Should entropy-off MM/GBSA be allowed as the default reporting surface for your use case?
  • Should multi-pose, multi-seed, or replicate averaging be required before signoff?
  • Are protonation and parameterization decisions explicit enough for scientific review?
